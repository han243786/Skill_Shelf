# Work Mode Router

拓扑式治理最重要的入口规则是：每次任务先判定工作模式。

不要把所有事情都叫“继续”。继续什么，必须先说清楚。

## 1. 四种入口

| 模式 | 什么时候用 | 改变什么 |
| --- | --- | --- |
| 重构模式 | 行为不变，只改变结构 | 既有节点、边、父级 facade、旧路径 |
| 推进模式 | 新增能力、新端口、新节点 | 新节点、新 route/schema/event/capability |
| 切面打磨模式 | 优化一个横向切面 | 复制影响范围，优化镜像，裁剪旧切面 |
| 文档债清理 | 修旧路径、旧索引、旧说明 | 地图准确性，不做功能开发 |

## 2. 入口卡

```text
work_mode:
reason:
allowed_scope:
topology_slice:
exit_gate:
```

字段解释：

| 字段 | 要求 |
| --- | --- |
| `work_mode` | 四种模式之一 |
| `reason` | 为什么不是其他模式 |
| `allowed_scope` | 本轮允许触碰的文件、节点、边、测试和文档 |
| `topology_slice` | 本轮影响的父节点、子节点、public surface 和旧新关系 |
| `exit_gate` | 最小可验证收口条件 |

## 3. 重构模式

目标：结构变清楚，行为保持等价。

必须回答：

- 原节点是什么？
- 新节点是什么？
- 父级 facade 如何保持兼容？
- public surface 是否变化？
- 哪些测试证明行为不变？
- 旧路径何时退役？

适合调用 `modular-refactor`。

## 4. 推进模式

目标：新增能力进入系统，而不是随手补丁。

必须回答：

- 新能力接哪个父级？
- 新增哪个端口、schema、event、route 或 UI surface？
- 需要新增模块节点吗？
- 旧能力是否受影响？
- 测试和拒绝行为是什么？

推进模式可以局部跳转到重构模式或切面打磨模式，但必须写返回条件。

## 5. 切面打磨模式

目标：优化一个横向切面，避免直接污染旧系统。

四段式：

```text
隔离 -> 分叉 -> 优化 -> 裁剪
```

含义：

1. 隔离：划定影响范围，标出所有会受影响的端口。
2. 分叉：复制受影响区的旧代码和治理资产，形成镜像拓扑。
3. 优化：只改镜像，直到新代码可独立运行。
4. 裁剪：切换到新拓扑，抛弃旧路径。

这对代码、模块树、全量树、契约、测试和文档同步生效。

## 6. 文档债清理

目标：修地图，不顺手做功能。

允许做：

- 确认旧路径是否存在。
- 删除、替换或归档旧引用。
- 更新索引。
- 跑对应门禁。

禁止做：

- 顺手重构代码。
- 顺手新增能力。
- 把历史债务混入当前功能 closeout。

## 7. 升档触发器

轻量任务命中任一条件，必须升档：

1. 改 public API、route、schema、event、capability。
2. 改状态、锁、持久化、事务、权限、安全边界。
3. 改模块父子关系或 public owner。
4. 删除旧代码、旧入口、旧契约、旧测试或旧当前态文档。
5. 影响多个父级模块。
6. 无法一句话说明 rollback。
