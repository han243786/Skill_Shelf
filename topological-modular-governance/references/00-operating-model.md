# Operating Model

拓扑式模块化治理的核心，是把项目从“文件集合”提升为“白箱网络”。

在普通项目里，人们常问：

```text
这个功能在哪个文件？
```

在拓扑式治理里，必须继续追问：

```text
这个文件属于哪个模块节点？
这个节点归哪个父级所有？
它通过哪条边和外界通信？
这次变更是在改节点、改边、改端口，还是只是在修地图？
```

## 1. 基本定义

| 概念 | 含义 |
| --- | --- |
| 节点 | 一个可审计的模块白箱，说明输入、输出、owner、public 方法、状态、副作用和测试 |
| 边 | 模块之间的通信路径，包括父级 facade、契约、adapter、事件、注册表或 public API |
| 父级 | 管理一组子节点的路由层、汇聚层或 facade 层 |
| 叶子 | 当前不值得继续拆分，或继续拆分会降低治理收益的最小治理单元 |
| 全量树 | 物理地图，回答项目里有什么文件 |
| 模块树 | 逻辑拓扑，回答模块如何归属、通信和被验证 |
| 工作模式 | 本轮如何改变拓扑：重构、推进、切面打磨或文档债清理 |

## 2. 精妙之处

这套治理真正厉害的地方，不是多写几份文档，而是把混乱开发中的隐性问题显性化。

### 2.1 它把 AI 幻觉变成可定位问题

AI 最容易在大项目里犯三种错：

1. 以为某个文件还存在。
2. 以为某个模块可以直接调用另一个模块。
3. 以为测试通过就代表结构没有退化。

拓扑式治理要求每个结论都落到真实节点、真实边、真实文件、真实 public 方法和真实门禁上。只要其中一项说不清，幻觉就会很快暴露，而不是在后续十几轮里越滚越大。

### 2.2 它让扩张变成“接入网络”，不是“堆代码”

新增能力时，不是把代码随便塞进一个地方，而是先回答：

```text
接哪个父级？
新增哪个端口？
是否新增子节点？
谁拥有 public surface？
旧入口如何共存或退役？
```

这样项目越大，越像一张有层级、有路由、有边界的网络，而不是一团互相引用的文件。

### 2.3 它允许快，但要求知道为什么能快

同父级下的独立子叶可以批量处理；纯 helper、DTO、映射、validation 可以轻量化。但状态机、锁、持久化、外部 API、公共协议必须升档。

提速来自“重复治理动作批量化”，不是来自“省略证明”。

### 2.4 它把性能捷径放到正确阶段

开发态优先父子通信，哪怕牺牲一点性能。这样结构可读、可测、可回退。

发布过渡时，如果开发者明确声明可以为了性能优化拓扑，才允许提出横向连接、旁路缓存或热路径直连。AI 不主动提出进入这个阶段，避免过早把结构复杂度引入开发态。

## 3. 四条核心不变量

1. **节点可解释**: 每个关键模块都能说明输入、输出、owner、public 方法、状态、副作用和测试。
2. **边可审计**: 模块通信必须走登记过的父级、契约、adapter、事件或接口。
3. **模式先行**: 每次任务先判定工作模式，不能把“继续”默认当成递归重构。
4. **收口一致**: closeout 要证明代码、模块树、全量树、契约、测试、文档和旧路径状态一致。

## 4. 最小拓扑卡

```text
work_mode:
parent_node:
affected_child_nodes:
changed_edges:
public_methods_or_ports:
state_or_side_effects:
full_tree_paths:
module_tree_nodes:
gates:
closeout_condition:
```

这张卡不要求长，但必须让下一个人知道本轮在网络上动了哪里。
