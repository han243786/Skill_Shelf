# Modular Refactor

这是一套给 vibecoding 使用的模块化重构方案。目标是让任何技术栈、任何规模的代码库，都能在合适的速度下完成高质量拆分、整理和重构，同时避免回退、冲突、AI 幻觉和依赖纠缠。

## 一句话说明

把一个复杂项目当成一座城市来治理：先画地图，再划区，再修路，最后逐片验收。不能让每个房间私拉电线，所有跨区通信都要先经过楼层或主干道。

## 它解决什么问题

模块化重构最怕三件事：

1. 改着改着不知道自己改到了哪里。
2. 拆出来的模块互相横向依赖，最后变成新的混乱。
3. AI 看起来推进很快，但悄悄漏掉行为、接口、测试或状态边界。

这套方案用三件事压住风险：

1. 模块树：先定义父模块、子模块、叶子模块。
2. 白箱节点：每个模块都要说明输入、输出、处理者、公共方法、状态和副作用。
3. 等价基线：每次拆分前后都要证明功能没有变。

## 通俗类比

你可以把项目想象成：

```text
城市
  -> 区
    -> 街道
      -> 建筑
        -> 房间
```

一个房间要联系另一个房间，不能自己从窗户拉线过去。它要先走建筑的楼道，再走街道，再到对方建筑。这就是父子通信规则。

这样做会让开发期慢一点，但项目不会乱。只有当开发者明确说“现在进入发布版本性能优化”时，才允许提出少量横向直连方案，用性能换结构复杂度。

## 核心流程

1. 画全局地图：列出顶层模块和剩余大文件。
2. 选择一个父模块：一次只治理一个父级边界。
3. 切子叶：按输入、输出、状态、公共方法和副作用划分叶子。
4. 建等价基线：记录原行为、接口、测试、错误信息和边界。
5. 抽离：把功能从旧文件移动到新模块，父级保留 facade。
6. 整理：让新模块内部更清楚，不改变外部行为。
7. 判断是否继续细分：值得就递归，不值得就 closeout。
8. 父级 closeout：确认这个父模块完成治理，再换下一个父模块。

## 三种执行强度

轻量：小模块、小函数、小 DTO，只做必要基线和快速检查。

标准：普通业务模块，走完整 baseline、抽离、整理、closeout。

重型：涉及状态机、持久化、锁、并发、公共 API、协议、外部 IO，必须慢走，不能为了速度跳过证明。

## 怎么提速

提速不是把规范砍掉，而是把重复劳动批量化。

最重要的加速方式是“同父级子叶并行波次”：

```text
父模块
  -> 子叶 A
  -> 子叶 B
  -> 子叶 C
```

如果 A、B、C 都归同一个父模块管理，并且彼此没有直接依赖，就可以同时准备它们的白箱、基线和子文件。真正修改父级 facade 的时候，仍然由一个 coordinator 串行合并，避免冲突。

适合并行的：

- DTO 家族
- validation 家族
- pure helper 家族
- registry/table/schema 映射
- 同质 route handler

不适合并行的：

- 共享 mutable state
- 数据库写入
- 锁和并发
- 状态机转移
- 公共协议和外部 API
- 多个子叶都要大改同一个父文件

## 最常用的执行口令

给 AI 的简单口令：

```text
使用 modular-refactor 方案。
先只读扫描项目，建立模块树和剩余大文件压力图。
选择一个父模块，冻结白箱边界和等价基线。
按抽离、整理、重构三步走。
允许同父级独立子叶做并行波次，但父级 facade 必须串行合并。
禁止子模块横向连接，除非我明确宣布进入发布版本性能优化。
每完成一个可验证步骤，给出 closeout 和下一步建议。
```

Rust 本地工作区专用口令：

```text
使用 modular-refactor 的 Rust 本地协议。
先扫描 Cargo workspace、crate facade、最大 .rs 文件、pub use/mod 树和状态/IO 边界。
按 crate -> parent module -> child leaf 的顺序推进。
保持 src/lib.rs 和 parent facade 的 public surface 稳定。
优先抽离 DTO、error、pure helper、validation、mapping/lowering。
runtime state、锁、异步生命周期、IO、协议和 public API 必须走重型模式。
允许同父级 Rust 子叶并行准备 child 文件，但 mod/pub use/facade 合并必须由一个 coordinator 串行完成。
每个叶子 closeout 必须报告 visibility 变化、parent-owned helper、cargo gate 和是否继续细分。
```

## 成功标准

一个模块真正完成，不是“代码挪出去了”，而是：

- 父级 facade 清楚。
- 子模块输入输出清楚。
- 关键 public 方法归属清楚。
- 状态和副作用归属清楚。
- 外部行为等价。
- 测试或 smoke 证明仍然成立。
- 模块树和白箱文档能指导下一个开发者继续推进。
